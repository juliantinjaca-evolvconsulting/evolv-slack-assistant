Slack App Setup Guide

Step-by-step instructions for creating and configuring your Sales Assistant Slack App

This guide walks you through the process of creating a Slack App that connects to Snowflake Cortex Agents. By the end of this guide, you will have:

  • A Slack App installed in your workspace
  • A Bot Token (xoxb-...) for API authentication
  • An App-Level Token (xapp-...) for Socket Mode connection
  • A User Group ID (S...) to control who can access the bot

These are required configuration items for running the Sales Assistant bot.

Prerequisites

What you need before getting started

1. Slack Workspace Access

You need admin access to a Slack workspace where you want to install the app. If you don't have admin access, contact your workspace administrator.

2. Manifest File

You need the manifest.json file from the project repository. This file contains all the app configuration (permissions, events, features).

3. Slack Account

You must be logged into Slack with the account that has admin permissions for the target workspace.

1Login to Slack API

Access the Slack App management portal

Navigate to Slack API Portal

Open your browser and go to the Slack API applications page:

https://api.slack.com/apps

Sign In

If you're not already logged in, Slack will prompt you to sign in. Use the account that has admin permissions for the workspace where you want to install the app.

Note: Make sure you're signing into the correct Slack workspace. If you belong to multiple workspaces, you can switch between them from the top-right dropdown after logging in.

View Your Apps

Once logged in, you'll see the "Your Apps" dashboard. This page lists all Slack apps you've created or have access to manage.

2Create App from Manifest

Use the manifest file to create a pre-configured app

Start App Creation

From the "Your Apps" page, click the "Create New App" button (usually green, top-right corner).

Choose Creation Method

You'll see two options. Select:

From an app manifest

This option allows you to create an app with all configurations pre-set from a JSON file.

Select Your Workspace

Choose the workspace where you want to install the app from the dropdown menu, then click "Next".

Important: You must have admin permissions in the selected workspace to complete the installation.

Paste the Manifest

You'll see a text editor with two tabs: YAML and JSON. Make sure JSON is selected, then:

  1. Open the manifest.json file from your project
  2. Copy the entire contents
  3. Paste it into the editor (replacing any existing content)
  4. Click "Next"

Review and Create

Slack will show you a summary of the app configuration:

  • App Name: Sales Assistant
  • Bot Scopes: assistant:write, chat:write, files:write, etc.
  • Socket Mode: Enabled
  • Events: message.im, assistant_thread_started, etc.

Review the settings, then click "Create".

Success! Your app is now created. You'll be redirected to the app's "Basic Information" page.

3Get the Bot Token

Install the app and retrieve the Bot User OAuth Token

Navigate to OAuth & Permissions

From your app's settings page, look at the left sidebar and click on:

OAuth & Permissions

Install App to Workspace

At the top of the page, you'll see an "Install to Workspace" button. Click it.

Slack will show you a permission request screen listing all the scopes your app is requesting. Review them and click "Allow".

Copy the Bot Token

After installation, you'll be redirected back to the OAuth & Permissions page. You'll now see a "Bot User OAuth Token" section with a token that starts with:

Bot User OAuth Token

This token is used for API calls to Slack (sending messages, uploading files, etc.)

xoxb-XXXXXXXXXXXX-XXXXXXXXXXXX-XXXXXXXXXXXXXXXXXXXXXXXX

Click the "Copy" button next to the token and save it securely.

Security Warning

This token provides access to your Slack workspace. Never commit it to version control or share it publicly. Store it securely and use environment variables to pass it to your application.

4Create App-Level Token

Generate a token for Socket Mode connections

Navigate to Basic Information

From the left sidebar, click on:

Basic Information

Find App-Level Tokens Section

Scroll down on the Basic Information page until you find the "App-Level Tokens" section.

Generate New Token

Click the "Generate Token and Scopes" button.

A modal will appear asking you to:

  1. Name your token: Enter a descriptive name (e.g., "Socket Mode Token")
  2. Add scopes: Click "Add Scope" and select connections:write
  3. Click "Generate"
Required Scope: The connections:write scope is essential for Socket Mode. Without it, your bot won't be able to establish a WebSocket connection to Slack.

Copy the App-Level Token

After generation, you'll see your new token. It starts with:

App-Level Token

This token is used for establishing Socket Mode connections

xapp-X-XXXXXXXXXXXX-XXXXXXXXXXXXX-XXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXX

Click the "Copy" button and save it securely. You can only see this token once, so make sure you copy it now!

One-Time Display

The App-Level Token is only displayed once after generation. If you lose it, you'll need to generate a new one. Make sure to copy and store it immediately.

5Configure User Group

Restrict bot access to authorized users only

The Sales Assistant bot uses Slack User Groups to control access. Only users who belong to the configured User Group will be able to interact with the bot. This is a security measure to ensure that only authorized personnel can query your sales data.

Create a User Group (if needed)

If you don't already have a User Group for authorized users, you'll need to create one.

Option A: From Slack Desktop/Web

  1. In Slack, click on "More" in the left sidebar
  2. Select "People & user groups"
  3. Click the "User Groups" tab
  4. Click "New User Group"
  5. Fill in the details:
    • Name: e.g., "Sales Assistant Users"
    • Handle: e.g., "sales-assistant-users"
    • Purpose: e.g., "Users authorized to use the Sales Assistant bot"
  6. Add the users who should have access
  7. Click "Create"
Documentation: For detailed instructions on creating and managing User Groups, see the official Slack Help Center article.

Get the User Group ID

Once you have a User Group, you need to obtain its ID. The easiest way is through the Slack web interface:

  1. Open Slack in your web browser (not the desktop app)
  2. Navigate to People & user groups
  3. Click on the User Group you want to use
  4. Look at the URL in your browser's address bar

The URL will look something like this:

https://app.slack.com/client/TXXXXXXXX/browse-user-groups/user_groups/SXXXXXXXXXX

The User Group ID is the part that starts with S at the end of the URL.

User Group ID

This ID identifies the group of users authorized to use the bot

SXXXXXXXXXX

Copy this ID — you'll need it for the environment configuration.

How Access Control Works

When a user sends a message to the bot:

  1. The bot checks if the user belongs to the configured User Group
  2. If yes: the message is processed normally
  3. If no: the user receives a denial message and the request is blocked

Important Considerations

Adding/Removing Users: Changes to User Group membership take effect within 2 minutes (the bot caches the member list for performance).

Group Management: Only workspace admins or designated group managers can add/remove members from the User Group.

6Generate RSA Keypair

Create authentication keys for Snowflake

Why RSA Keys?

Snowflake service accounts use RSA key-pair authentication instead of passwords. This provides:

  • Enhanced Security: No passwords to store or rotate
  • Automation-Friendly: Ideal for service accounts and CI/CD pipelines
  • Non-Interactive: No MFA prompts for automated processes

Generate the Key Pair

Open your terminal and run these commands to generate a 2048-bit RSA key pair:

Terminal - Generate Private Key 
# Generate private key in PKCS#8 format (required by Snowflake)
openssl genrsa 2048 | openssl pkcs8 -topk8 -inform PEM -out rsa_key.p8 -nocrypt
Terminal - Extract Public Key 
# Extract the public key from the private key
openssl rsa -in rsa_key.p8 -pubout -out rsa_key.pub

Verify Your Keys

You should now have two files in your current directory:

Generated Files

  • rsa_key.p8 — Private key (keep this SECRET, never commit to git)
  • rsa_key.pub — Public key (this will be added to Snowflake)

View the Public Key

Display your public key - you'll need this content for the next step:

Terminal 
cat rsa_key.pub

The output will look like this:

Example Output 
-----BEGIN PUBLIC KEY-----
MIIBIjANBgkqhkiG9w0BAQEFAAOCAQ8AMIIBCgKCAQEA...
...multiple lines of base64 encoded content...
...AQAB
-----END PUBLIC KEY-----

Security Best Practices

Private Key: Add rsa_key.p8 to your .gitignore file immediately. Never commit private keys to version control.

Storage: Store the private key in a secure location (e.g., secrets manager, encrypted vault) for production deployments.

7Create Service Account

Set up a dedicated service user for the application

Why a Service Account?

A service account (TYPE = SERVICE) is a special type of Snowflake user designed for applications:

  • No Password: Cannot login via UI, only programmatic access
  • Audit Trail: All queries are tracked under a dedicated identity
  • Principle of Least Privilege: Grant only necessary permissions

Create the Service User

Run the following SQL in Snowflake (requires ACCOUNTADMIN or USERADMIN role):

SQL - Create Service Account 
-- Create the service account with RSA key authentication
CREATE USER SVC_SALES_ASSISTANT
    TYPE = SERVICE
    RSA_PUBLIC_KEY = 'MIIBIjANBgkqhkiG9w0BAQEFAAOCAQ8AMIIBCgKCAQEA...'
    DEFAULT_WAREHOUSE = 'COMPUTE_WH'
    DEFAULT_ROLE = 'SALES_ASSISTANT_ROLE'
    COMMENT = 'Service account for Sales Assistant Slack bot';

Important: Public Key Format

When copying your public key content, you must remove the header and footer lines:

  • Remove: -----BEGIN PUBLIC KEY-----
  • Remove: -----END PUBLIC KEY-----
  • Remove: All line breaks (paste as a single continuous string)

Grant Required Permissions

Grant the service account the role that has access to your Cortex Agent:

SQL - Grant Role 
-- Grant the appropriate role to the service account
GRANT ROLE SALES_ASSISTANT_ROLE TO USER SVC_SALES_ASSISTANT;

-- Ensure the role has access to necessary objects
GRANT USAGE ON DATABASE UNIFAI TO ROLE SALES_ASSISTANT_ROLE;
GRANT USAGE ON SCHEMA UNIFAI.SALES_COPILOT TO ROLE SALES_ASSISTANT_ROLE;
GRANT USAGE ON CORTEX AGENT UNIFAI.SALES_COPILOT.SALES_COPILOT_VNEXT 
    TO ROLE SALES_ASSISTANT_ROLE;

Verify the Service Account

Confirm the user was created correctly:

SQL - Verification 
-- Check user details
DESCRIBE USER SVC_SALES_ASSISTANT;

-- Verify RSA key is configured
SELECT name, type, has_rsa_public_key 
FROM SNOWFLAKE.ACCOUNT_USAGE.USERS 
WHERE name = 'SVC_SALES_ASSISTANT';

8Create Thread Table

Store conversation context between Slack and Cortex Agent

Why Thread Mappings?

The Sales Assistant maintains conversation context by mapping Slack thread IDs to Snowflake Cortex Agent thread IDs. This allows:

  • Conversation Continuity: The agent remembers previous messages in a thread
  • Context Awareness: Follow-up questions understand prior context
  • Multi-User Support: Each user's conversations are tracked independently

Create the Table

Run this SQL to create the thread mappings table:

SQL - Create Thread Mappings Table 
-- Create table to store Slack-to-Snowflake thread mappings
CREATE TABLE IF NOT EXISTS SLACK_THREAD_MAPPINGS (
    SLACK_THREAD_ID     VARCHAR PRIMARY KEY,
    SNOWFLAKE_THREAD_ID VARCHAR,
    LAST_MESSAGE_ID     NUMBER,
    USER_ID             VARCHAR,
    CREATED_AT          TIMESTAMP_NTZ DEFAULT CURRENT_TIMESTAMP(),
    UPDATED_AT          TIMESTAMP_NTZ DEFAULT CURRENT_TIMESTAMP()
);

-- Add comment for documentation
COMMENT ON TABLE SLACK_THREAD_MAPPINGS IS 
    'Maps Slack thread IDs to Snowflake Cortex Agent thread IDs for conversation continuity';

Grant Permissions

Ensure the service account can read and write to this table:

SQL - Grant Table Permissions 
-- Grant SELECT, INSERT, UPDATE on the thread mappings table
GRANT SELECT, INSERT, UPDATE ON TABLE SLACK_THREAD_MAPPINGS 
    TO ROLE SALES_ASSISTANT_ROLE;

Table Schema Explanation

Column Purpose
SLACK_THREAD_ID Unique identifier from Slack (the thread timestamp)
SNOWFLAKE_THREAD_ID UUID returned by Cortex Agent API
LAST_MESSAGE_ID Tracks the last processed message (for deduplication)
USER_ID Slack user ID who initiated the thread

10Environment Variables

Set up all required configuration for the application

Required Environment Variables

The Sales Assistant requires the following environment variables to be configured:

.env File Template 
# ============================================
# SLACK CONFIGURATION
# ============================================
SLACK_BOT_TOKEN=xoxb-your-bot-token
SLACK_APP_TOKEN=xapp-your-app-level-token
ALLOWED_USER_GROUP=S0123456789

# ============================================
# SNOWFLAKE CONFIGURATION
# ============================================
SNOWFLAKE_ACCOUNT=your-account-identifier
SNOWFLAKE_USER=SVC_SALES_ASSISTANT
SNOWFLAKE_PRIVATE_KEY_PATH=./rsa_key.p8
SNOWFLAKE_WAREHOUSE=COMPUTE_WH
SNOWFLAKE_DATABASE=UNIFAI
SNOWFLAKE_SCHEMA=SALES_COPILOT
SNOWFLAKE_ROLE=SALES_ASSISTANT_ROLE

# ============================================
# CORTEX AGENT CONFIGURATION
# ============================================
CORTEX_AGENT_NAME=SALES_COPILOT_VNEXT

Where to Find Each Value

Slack Values (from previous steps)

Variable Source
SLACK_BOT_TOKEN OAuth & Permissions → Bot User OAuth Token (STEP 4)
SLACK_APP_TOKEN Basic Information → App-Level Tokens (STEP 4)
ALLOWED_USER_GROUP User Group URL → ID starting with S (STEP 5)

Snowflake Values

Variable How to Find
SNOWFLAKE_ACCOUNT Your Snowflake URL: https://<account>.snowflakecomputing.com
SNOWFLAKE_USER The service account created in STEP 7
SNOWFLAKE_PRIVATE_KEY_PATH Path to rsa_key.p8 generated in STEP 6

For Docker Deployments: Base64 Encode the Private Key

When deploying with Docker or in cloud environments, it's often easier to pass the private key as an environment variable rather than mounting a file:

Terminal - Base64 Encode Private Key 
# Encode the private key as a single-line base64 string
cat rsa_key.p8 | base64 | tr -d '\n'

Then use this variable instead:

Environment Variable for Docker 
# Use base64-encoded key instead of file path
SNOWFLAKE_PRIVATE_KEY_BASE64=LS0tLS1CRUdJTiBQUklWQVRFIEtFWS0tLS0t...

Security Reminder

Never commit .env files to version control. Add .env to your .gitignore.

Production deployments: Use your cloud provider's secrets manager (AWS Secrets Manager, Azure Key Vault, GCP Secret Manager, etc.)

11Run the Application

Start the Sales Assistant locally or deploy to the cloud

Option A: Run Locally with Python

For development and testing, you can run the application directly:

Terminal - Local Execution 
# Install dependencies
pip install -r requirements.txt

# Run the application
python app.py

You should see output similar to:

Expected Output 
⚡️ Bolt app is running!
🔌 Connected to Slack via Socket Mode
✅ Snowflake connection established
🤖 Sales Assistant is ready to receive messages

Option B: Run with Docker

For a containerized deployment:

Terminal - Docker Build & Run 
# Build the Docker image
docker build -t sales-assistant .

# Run with environment file
docker run --env-file .env sales-assistant

# Or run with individual environment variables
docker run \
  -e SLACK_BOT_TOKEN=xoxb-... \
  -e SLACK_APP_TOKEN=xapp-... \
  -e ALLOWED_USER_GROUP=S... \
  -e SNOWFLAKE_ACCOUNT=... \
  -e SNOWFLAKE_USER=SVC_SALES_ASSISTANT \
  -e SNOWFLAKE_PRIVATE_KEY_BASE64=... \
  sales-assistant
